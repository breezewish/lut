# RAW Alchemy interface critique

Target: `web/src/App.tsx` and its rendered application shell.

## Executive assessment

The pre-fix interface scored **23/40** on Nielsen's heuristics. Its desktop comparison workspace was focused, restrained, and low in AI-slop risk. The primary weaknesses were task order on mobile, recovery from damaged RAW files, and insufficient feedback during sequential batch export.

The visual system appropriately used one familiar product sans, flat surfaces, limited magenta emphasis, neutral image canvases, and no decorative motion. The comparison itself was the strongest part of the journey.

## Priority findings

### P1 — Mobile task order hid the primary journey

At 390 × 844, the first actionable file picker was below the fold, processing controls consumed about 606px before a file existed, `Add RAWs` was hidden, and export appeared before comparison after loading. The sidebar drop zone looked actionable but only accepted drag-and-drop.

Resolution: processing controls are progressively disclosed after selection; mobile retains Add RAW; the empty sidebar drop target is hidden on narrow screens; the desktop drop target is clickable; DOM order is edit, compare, then export; the 35rem flex-growth defect is removed.

### P1 — RAW failures exposed implementation details and allowed invalid export

The rendered error exposed `LibRaw std::runtime_error` and an internal code. A failed selected export was disabled, but batch export could still be offered without explaining what would be skipped.

Resolution: known LibRaw failures map to a product-level damaged-or-unsupported message; the error surface offers Remove file and Choose another RAW; failed files show consistent metadata and a dedicated empty preview; batch export filters known failures and reports skipped or newly failed files.

### P1 — Batch export lacked progress, control, and a clear ending

The UI only showed an indeterminate `Exporting…` label and a pulsing dot. It did not expose the current file, `n/N`, partial success, or a way to stop remaining work.

Resolution: sequential export reports the current file and position, exposes Stop after current, continues past per-file failures, uses distinct textual queue states, and leaves a persistent success/partial/stopped summary after download.

### P2 — The LUT output assumption was hidden as tiny footer text

The unverified output profile is a decision-relevant color caveat, but it was shown in a low-contrast 10.24px footer beside the internal term `Corrected v2`.

Resolution: every selected Look now carries an always-visible statement that the output profile is undeclared and the preview assumes sRGB. The footer uses 12px readable product language and removes the internal version term.

### P2 — Twenty-seven looks exceeded a plain Select's scan limit

The menu exposed 27 options and up to 15 at once, with a long Fujifilm group and no search or recent choices.

Resolution: a text search filters across brand and look name, and up to four recent choices appear as the first group. The compact grouped Select remains the final chooser.

## Secondary findings and resolutions

- Single-file state exposed duplicate `Export selected` and `Export all` actions. `Export all` now appears only for a true batch.
- Queue deletion was hover-dependent and irreversible. Remove is visible, queue text names every status, and remove/clear actions expose Undo.
- Exposure lacked direct numeric entry. The slider remains, with a bounded numeric EV field and Reset.
- Ready and exported states used the same green dot. They now have distinct colors and explicit text.
- Queue failure metadata contradicted itself. It now says `Unable to decode`.
- Export mutations could race with queue removal. Clear and remove are disabled during export while Stop after current remains available.
- The primary color used white 12px text at marginal contrast. Primary surfaces now use dark text.
- The queue helper was 11.52px. It and other key microcopy are normalized to at least 12px, with stronger muted contrast.

## Detector findings

The source detector returned zero findings. Browser detector warnings were evaluated individually:

- Brand low contrast: the detector misread the OKLCH background as white, but the underlying small-text contrast was still marginal and was fixed.
- Queue helper tiny text: valid and fixed.
- Overused or single Inter-compatible font: false positive for a dense product tool; the design contract intentionally uses one familiar system sans.
- Flat type hierarchy: the rule over-weighted compact utility labels; image scale, layout, weight, and surface hierarchy remain clear. Fragmented micro-sizes were nevertheless normalized where they impaired readability.

## Journey and persona outcomes

The arrival now foregrounds local import and privacy before processing choices. Loading retains the camera JPEG and explicit decode status. Comparison remains the emotional high point. Export now provides reassurance and control during the high-wait phase, while the persistent summary gives the journey a clear end. Failure is recoverable without implementation language.

Power users gain search, recent looks, direct EV entry, explicit batch progress, and stop-after-current. First-time users see product language and actionable recovery. Distracted mobile users retain a visible Add RAW action and encounter comparison before export.

## Final verdict

All actionable P1/P2 findings and valid detector findings were addressed. The remaining detector font warnings are documented false positives consistent with the product-interface typography register.
