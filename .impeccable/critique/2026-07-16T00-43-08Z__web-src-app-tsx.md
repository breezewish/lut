## Anti-patterns verdict

**LLM assessment:** The interface does not read as AI-generated. It uses a restrained rose/neutral product palette, flat hierarchy, familiar controls, and an image-first comparison surface without decorative cards, glass, gradients, ornamental motion, or filler copy.

**Deterministic scan:** The final CLI layout scan and full `web/src` scan both returned `[]`; the arbitrary Tailwind value and z-index search also returned no matches. Browser injection on the final desktop and mobile export-error states reported the intentional single Inter family, compact product typography, and one contrast false positive on `.lut-assumption`. That amber text is intentionally emphasized on the dark raised control surface and is visually high contrast; the detector's warning does not match the rendered surface.

**Visual overlays:** Mutable `detect.js` injection succeeded in headless Chromium at 1440×1000 and 390×844. This unattended desktop task has no persistent user-visible Human tab, so the headless browser injection and full-page screenshots are the required fallback evidence.

## Overall impression

The application has a clear task sequence: queue context, compact recipe controls, dominant Base/LUT comparison, then one export decision. The final error state remains part of that workspace instead of replacing it, so a TIFF failure no longer looks like lost decode work.

## Layout assessment

- Desktop uses a 280 px queue rail and a flexible workspace; the two comparison panels share the available width without horizontal overflow at 1440 px.
- Mobile switches to a single-column flow, moves the compact app bar into a sticky control, and keeps all interactive controls at least 44 px rather than shrinking the desktop arrangement.
- The hierarchy follows queue → look/exposure → Base/LUT comparison → export result. The selected error wraps in `.selected-error`, so the real Worker message remains readable at 390 px.
- Spacing stays on the existing CSS scale, and the export action remains reachable in short or scrolling viewports.

## What's working

- Base and LUT previews remain visible after export failure, preserving the user's visual context and making retry credible.
- The queue summary says `Export failed · retry available`, while the detailed `role="alert"` contains the concrete Worker error instead of a generic fallback.
- Status is communicated through text and accessible roles, not color alone; error red remains an accent rather than becoming a full-page danger treatment.
- Desktop and mobile error-state screenshots have `scrollWidth === clientWidth`, confirming no horizontal overflow.
- Privacy, undeclared output-profile semantics, disabled corrections, batch progress, and stop behavior appear at the moment they are relevant.

## Priority issues

- **[P1 · resolved] Export failure was modeled as decode failure:** The previous shared error state hid a valid preview and prevented retry. Queue state now distinguishes `decode-error` from `export-error`; only decode failure invalidates the preview.
- **[P1 · resolved] Export failure hid the concrete cause:** The retained-preview message was generic. The selected result now exposes the actual Worker error in a wrapping alert while retaining the retry action.
- **[P2 · resolved] Empty workspace retained a large WASM renderer:** Removing the selected file or clearing the queue now sends an explicit `clear` command and frees the cached `PreviewRenderer`.

No unresolved P0, P1, or P2 issue remains in the final pass.
